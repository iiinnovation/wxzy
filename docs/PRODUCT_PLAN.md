# 中医学习微信小程序产品规划

## 1. 产品概述

### 1.1 产品定位

一款面向个人使用的中医学习工具。用户导入或维护自己的中医 PDF 文献，通过阅读、检索、问答、主动回忆和间隔复习，逐步建立对中医理论、方剂、药物和医案的理解。

产品不是在线问诊、自动诊断、处方或医疗决策系统。

### 1.2 核心价值

- 把大量 PDF 变成可检索、可引用的个人知识库。
- 把被动阅读转成主动回忆和重复练习。
- 用 Qwen 辅助解释难懂的古文和中医概念，但保留原文出处。
- 根据复习表现安排下一次学习时间，降低遗忘。
- 通过方剂、证候、病机和医案之间的关系帮助理解，而不只是背诵。

### 1.3 目标用户

首要用户是产品所有者本人，具有持续学习中医的需求，并拥有一批个人 PDF 文献。

第一阶段不以公开运营、多用户社区或商业化为目标。设计上保留未来扩展空间，但不为这些场景提前增加复杂度。

## 2. 产品目标和非目标

### 2.1 MVP 目标

在一套可控的中医 PDF 样本上完成以下闭环：

```text
导入文献 -> 检索和阅读 -> 生成或创建知识卡片 -> 主动回忆
-> 根据结果安排复习 -> 通过 Qwen 解释并回到原文核对
```

### 2.2 非目标

- 不做面向患者的在线诊疗。
- 不根据个人症状自动开方、推荐药物或替代医生判断。
- 不在第一版做社交、排行榜、课程市场和内容分发平台。
- 不追求一次性自动处理所有 PDF；先确保小批量文献的准确率。
- 不把模型生成内容直接当作权威知识，未经确认的内容不能进入正式复习库。

## 3. 核心学习闭环

### 3.1 学习流程

1. 用户选择书籍、章节或知识主题。
2. 阅读原文及现代解释。
3. 对知识点进行主动回忆，先作答再查看答案。
4. 用户标记掌握程度，系统记录结果。
5. 系统按照间隔复习规则生成下一次复习任务。
6. 复习时可让 Qwen 对错误、遗漏和概念混淆进行解释。
7. 用户可以跳回原文页码，核对模型答案。

### 3.2 复习间隔

MVP 使用简单、可解释的间隔：当天、1 天、3 天、7 天、15 天、30 天。复习结果调整下一次间隔：

- 完全掌握：延长间隔。
- 模糊或部分掌握：维持或小幅延长。
- 未记住：缩短间隔，并建议重新阅读相关原文。

后续可替换为 SM-2 或 FSRS，但复习记录和算法必须解耦，以便升级而不丢失历史数据。

### 3.3 学习对象

知识库中可复习的对象包括：

- 概念卡：病机、证候、治法、功效、禁忌。
- 方剂卡：组成、功效、主治、配伍和鉴别。
- 药物卡：性味、归经、功效和常见配伍。
- 对比卡：相近证候、方剂或药物的差异。
- 医案卡：症状、舌脉、辨证、治法、处方和按语。
- 古文卡：原文、关键词解释和现代语言转述。

## 4. 功能规划

### 4.1 MVP 功能

#### 文献库

- 书籍列表、作者、分类和处理状态。
- 章节树和页码信息。
- 文献内全文搜索。
- 原文阅读和引用片段展示。

#### 知识检索

- 关键词搜索。
- 自然语言问答。
- 混合检索、Rerank 和出处展示。
- 检索不到时明确提示依据不足。

#### 学习卡片

- 手动创建卡片。
- 根据章节生成候选卡片。
- 卡片审核、编辑、归档。
- 问题、答案、解析、标签、来源页码。

#### 每日复习

- 今日到期队列。
- 先回答后显示答案。
- 掌握程度评价。
- 下一次复习时间。
- 连续正确、遗忘和完成记录。

#### AI 学习助手

- 古文转现代语言。
- 分层解释概念。
- 根据当前片段出题。
- 批改用户答案并指出遗漏。
- 比较两个方剂、证候或药物。

#### 管理和任务状态

- PDF 上传。
- 文本抽取和 OCR 状态。
- 向量化状态。
- 失败重试。
- 模型生成内容审核。

### 4.2 后续功能

- 医案推理练习。
- 按主题的学习路径，例如脾胃、温病、伤寒、金匮。
- 跨书籍知识关联。
- 学习统计、遗忘曲线和薄弱知识点。
- 分包和预下载优化。
- 多设备同步。

## 5. 内容处理与知识库

### 5.1 文献处理流水线

```text
PDF 上传
  -> 文件校验和病毒扫描
  -> 文本抽取
  -> 扫描版 OCR
  -> 清理页眉、页脚和乱码
  -> 章节、标题、页码识别
  -> 中医语义分块
  -> 关键词索引和向量化
  -> 可检索状态
```

每个片段必须保留：文档名、作者、版本、章节、页码、原文、文本类型和处理版本。

### 5.2 分块原则

优先按书籍结构、章节标题和中医语义边界分块，而不是只按固定字数切割。病机、症状、治法、方剂和按语之间需要尽可能保留上下文。

扫描古籍、繁体字、竖排文字、表格和图片需要单独记录 OCR 质量，低质量片段不能直接用于自动生成正式卡片。

### 5.3 AI 生成内容审核

Qwen 生成的卡片、题目和解析先保存为“候选”，包含生成时使用的文献片段和模型版本。人工确认后才进入正式学习库。模型内容必须能回溯到原文；无法回溯的内容只能标记为待核对。

## 6. 技术方案

### 6.1 总体架构

```text
微信小程序
    |
    | HTTPS
    v
阿里云后端 API
    +-- 用户和学习进度
    +-- 文献和卡片服务
    +-- 检索编排服务
    +-- Qwen 调用服务
    +-- 导入和生成任务服务
    |
    +-- PostgreSQL + pgvector
    +-- Redis：队列、缓存、幂等控制
    +-- OSS：PDF、图片、OCR 中间文件
    +-- 向量模型和 Rerank 模型
```

### 6.2 服务边界

- 小程序：页面、交互、状态展示和轻量本地缓存。
- API 服务：鉴权、业务规则、检索编排和复习算法。
- 文献任务服务：抽取、OCR、分块、向量化和重试。
- AI 服务：提示词、上下文拼接、Qwen 调用、引用校验和限流。
- 数据层：关系数据、向量、文件和任务状态分离保存。

个人自用的第一版优先使用 PostgreSQL + pgvector，避免过早引入独立向量数据库。数据规模和检索压力上升后再评估 Milvus 或 Elasticsearch。

### 6.3 问答链路

```text
用户问题
  -> 问题清洗和关键词提取
  -> 关键词检索 + 向量检索
  -> 候选片段合并
  -> Rerank
  -> 选择带页码的上下文
  -> Qwen 生成
  -> 引用和风险检查
  -> 小程序展示答案
```

模型提示词必须要求：只依据上下文回答、区分原文和解释、禁止伪造出处、依据不足时明确说明，并对个人症状问题给出学习边界提示。

## 7. 数据实体草案

### Document

文献基本信息、文件地址、作者、分类、版本、处理状态和版权备注。

### DocumentChunk

文档片段、章节、页码、原文、文本类型、向量、关键词和处理版本。

### KnowledgeCard

问题、答案、解析、类型、标签、来源片段、审核状态和生成信息。

### ReviewRecord

用户、卡片、回答结果、掌握程度、复习时间、下次到期时间和算法版本。

### ChatRecord

问题、检索片段、回答、引用、模型版本、耗时和风险标记。敏感内容和完整原文不应无期限保存。

### ImportTask

文件、任务阶段、进度、错误、重试次数和处理版本。

## 8. 页面信息架构

MVP 建议使用四个主要入口：

1. **学习**：今日复习、继续阅读、薄弱知识点。
2. **文献**：书籍、章节、原文阅读和文献搜索。
3. **问答**：基于个人文献的问答和引用。
4. **我的**：学习统计、卡片管理、导入任务和设置。

关键页面包括：文献详情、章节阅读、知识卡片、复习答题、答案解析、问答详情和导入任务详情。

## 9. MVP 里程碑

### M0：样本和指标

- 选取 5～10 本具有代表性的 PDF。
- 确认哪些文献有版权或个人使用授权。
- 准备 30～50 个真实检索问题和标准出处。
- 定义检索命中、引用准确和生成内容审核指标。

### M1：知识库基础能力

- 完成上传、文本抽取、OCR、章节识别和页码保存。
- 完成分块、向量化、关键词检索和向量检索。
- 能根据问题返回正确文献片段。

### M2：小程序阅读和问答

- 完成文献列表、章节阅读和文献内搜索。
- 接入 Rerank 和 Qwen。
- 展示答案、引用和依据不足状态。

### M3：复习闭环

- 完成卡片创建、审核和编辑。
- 完成主动回忆、结果记录和间隔复习队列。
- 完成每日学习页和基础统计。

### M4：质量与稳定性

- 处理长文本、OCR 错误、网络超时和任务失败。
- 完成权限、日志脱敏、隐私说明和版权访问控制。
- 在微信开发者工具和真实手机上验证性能与兼容性。

## 10. MVP 验收标准

- 代表性 PDF 能成功解析，章节和页码可追溯。
- 真实问题的检索结果能返回相关原文，不依赖模型猜测。
- Qwen 回答的重要结论有文献引用；无依据时会明确提示。
- 用户可以完成“阅读 -> 作答 -> 查看解析 -> 安排复习”的完整流程。
- 复习结果会影响下一次到期时间，刷新或重新进入后状态不丢失。
- 失败任务可以重试，网络异常不会造成重复卡片或重复复习记录。
- 小程序前端不包含任何模型密钥、数据库凭据或服务端内部地址。

## 11. 风险与决策记录

### 主要风险

- OCR 错误导致方剂、药物和古文内容失真。
- 分块不合理导致检索缺少上下文。
- Qwen 生成错误或混淆相近方剂、证候。
- PDF 版权和个人信息处理不合规。
- 长文阅读和问答结果导致小程序包体积或首屏性能下降。

### 当前建议决策

- 先做个人单用户模型，不做公开内容平台。
- 先做结构化引用和复习闭环，再扩大自动生成能力。
- 先使用 PostgreSQL + pgvector，按真实数据规模决定是否引入 Milvus。
- 所有自动生成知识先审核，默认不直接进入复习队列。
- 所有医疗相关回答都保持学习工具边界，不输出个体诊疗结论。
