# wxzy 产品需求文档（PRD）

状态：Baseline v1，待按实施计划交付
日期：2026-07-22
产品形态：个人自用的中医学习微信小程序 + 独立文档处理控制面

## 1. 摘要

wxzy 帮助唯一使用者把自有中医 PDF 转换为可追溯、可审核的学习内容，并通过主动回忆、标准 FSRS、薄弱点练习和负荷控制形成长期学习习惯。

产品包含两个相互连接但不混合的部分：

1. **文档处理控制面**：在本地或服务器批量处理 PDF，产出结构化内容、候选卡和发布包。
2. **学习小程序**：只消费已发布内容，保存唯一用户的资料、目标、进度和复习记录。

文档处理是否完成以页覆盖和质量报告判断；小程序是否可用以用户能否完成一轮真实学习判断。两者不得使用同一个“完成”状态。

## 2. 背景与现状

当前原型已经具备：

- FastAPI + SQLite/PostgreSQL 基础服务。
- Bearer Token 单用户访问。
- 书籍、卡片、到期队列、四档评分和基础统计接口。
- 微信原生小程序的今日、复习、书库、我的页面。
- MinerU 两本书各 10 页的解析样本。
- 15 张 approved 示例卡和轻量 `fsrs-v1` 调度器。

当前原型不具备：

- 唯一用户身份、微信登录、个人学习档案和多设备同步。
- 内容目录与个人卡片进度分离。
- 经过验证的标准 FSRS 实现。
- 每日时间预算、新卡控制、未来负荷和薄弱点计划。
- 7 本 PDF、704 页的完整解析、结构化和发布覆盖。
- 可恢复的批处理、全量遍历、系统化质量门禁和审核工作流。
- 数据库迁移、自动化测试、API 版本化和 CI。

## 3. 产品愿景

使用者每天打开小程序后，不需要自行判断“今天学什么”，系统根据目标、可用时间、到期复习、薄弱知识和未来负荷给出一条清晰计划；每个学习结论都能回到正确的书籍、章节和页码核对。

## 4. 产品原则

### P-01 学习优先于功能数量

每个新增功能都必须改善理解、回忆、反馈、间隔或学习负荷中的至少一项。

### P-02 来源优先于模型表达

原文和人工审核是事实依据。模型只能生成候选内容、解释遗漏和提出卡片修复建议。

### P-03 全量内容不等于全量激活

7 本书应全部进入可检索、可生成的内容目录，但只有用户选择或计划引入的卡片才进入复习状态。

### P-04 单用户不等于无身份

系统需要一个稳定 Owner 和个人进度模型，但不建设注册、角色、租户、社交或公共内容体系。

### P-05 负荷必须可控

新卡不能无限增加。系统根据每日时间和未来复习量动态限制新内容。

### P-06 医疗边界明确

产品是学习工具，不收集症状形成健康档案，不提供个体诊断、处方或替代医生建议。

## 5. 用户与角色

### 5.1 Owner/Learner

唯一实际用户，既拥有内容和服务器，也在小程序中学习。其核心需求是：

- 保存学习目标、偏好和历史。
- 在手机上快速完成每日学习。
- 按书籍、章节和主题理解自己的进度。
- 发现薄弱知识和相似概念混淆。
- 核对原文，修正错误或低质量卡片。

### 5.2 Content Operator

与 Learner 是同一个人，但处于维护场景。通过本地 CLI 或服务端管理任务完成 PDF 处理、质量检查和候选卡审核。该角色不通过学习小程序上传大 PDF 或控制 MinerU。

## 6. 目标与非目标

### 6.1 产品目标

- 建立唯一 Owner 身份与完整个人学习档案。
- 处理 `docs/` 中 7 本 PDF 的全部 704 页。
- 建立文档、章节、内容块、候选卡、发布卡和来源关系。
- 提供以主动回忆和标准 FSRS 为核心的每日学习计划。
- 根据真实学习表现控制新卡、识别薄弱点并建议有效练习方式。
- 保持每张发布卡可回到文档版本、PDF 页和原文摘录。
- 在开发者工具和真机完成可持续使用的学习闭环。

### 6.2 明确非目标

- 多用户注册、组织、角色和租户权限。
- 社区、排行榜、好友、公共牌组和内容交易。
- 付费、广告和商业化运营。
- 小程序直接调用 MinerU、Qwen 或上传数百 MB PDF。
- 公开分享受版权保护的整本 PDF 或完整 OCR。
- 面向患者的症状采集、诊断和处方。
- 第一阶段构建通用笔记编辑器或完整 PDF 阅读器。

## 8. 核心用户旅程

### 8.1 首次使用

1. 用户通过微信身份登录，首次 OpenID 绑定为唯一 Owner。
2. 用户设置学习目标、目标日期、每日可用分钟、每周学习日和科目优先级。
3. 系统展示 7 个学科及内容覆盖，不展示 MinerU 批次或本地路径。
4. 用户选择起始学科或接受推荐顺序。
5. 系统建立首周保守计划，不在没有历史时伪装成高度个性化。

### 8.2 每日学习

1. 今日页展示预计时长、到期复习、新内容和薄弱练习。
2. 用户开始会话，系统优先安排到期卡。
3. 用户先主动回忆，再显示答案和来源。
4. 用户评分，系统记录评分、耗时、提示和会话上下文。
5. 若仍有时间且未来负荷可控，系统引入少量新卡。
6. 会话结束后展示完成量、主要错误和下一步，不制造惩罚性连续打卡压力。

### 8.3 新章节学习

1. 用户在学科页选择章节。
2. 系统显示章节内容覆盖、可学习卡数和先修建议。
3. 用户将章节或部分主题加入学习。
4. 系统按每日预算逐步引入卡片，不一次性创建全部到期状态。

### 8.4 薄弱点修复

1. 系统发现重复遗忘、耗时过长或相近概念混淆。
2. 系统建议返回原文、拆小卡片、做对比题或做无提示书写题。
3. 用户确认后产生修复任务或新的候选卡。
4. 修复结果继续保留来源和生成记录。

### 8.5 内容维护

1. Operator 在控制面查看文档批次和质量报告。
2. 对低风险章节抽样，对高风险卡逐张复核。
3. 通过审核的内容生成版本化发布包。
4. 后端幂等导入发布包；小程序随后看到新内容目录。

## 9. 功能需求

### 9.1 身份与会话

- **AUTH-001**：系统只能存在一个 active Owner。
- **AUTH-002**：正式环境使用微信 `wx.login` 换取服务端 Session。
- **AUTH-003**：首次成功登录绑定 Owner；其他 OpenID 默认拒绝，除非执行显式重绑定流程。
- **AUTH-004**：开发环境可启用固定 Token，但必须与生产认证隔离。
- **AUTH-005**：Session 可撤销、过期和刷新，OpenID 不返回给普通业务页面。

### 9.2 学习档案

- **PROFILE-001**：保存目标类型、目标日期、每日分钟数和每周学习日。
- **PROFILE-002**：保存学科优先级、初始自评和可选显示昵称。
- **PROFILE-003**：支持修改目标，修改后重新生成未来计划但不删除历史。
- **PROFILE-004**：支持导出和删除个人学习数据。

### 9.3 内容目录

- **CAT-001**：展示 7 本书、章节、可学习卡数量和内容版本。
- **CAT-002**：展示卡片问题、答案、类型、标签和审核状态。
- **CAT-003**：发布卡必须展示来源书籍、章节、PDF 页和原文摘录。
- **CAT-004**：支持按书籍、章节、类型和关键词搜索发布内容。
- **CAT-005**：小程序不得访问 raw OCR、解析压缩包、本地文件路径或处理密钥。
- **CAT-006**：内容版本更新不能静默改写既有学习历史。

### 9.4 加入学习

- **ENROLL-001**：用户可将书籍、章节、主题或单张发布卡加入学习。
- **ENROLL-002**：加入学习只创建 enrollment，不立即把全部卡设为当天到期。
- **ENROLL-003**：用户可暂停、恢复或退出某个学习范围，历史记录保留。
- **ENROLL-004**：新卡引入顺序考虑章节顺序、先修关系和用户优先级。

### 9.5 每日计划

- **PLAN-001**：计划输出预计分钟、到期数、新卡数和薄弱练习数。
- **PLAN-002**：计划先满足到期复习，再按剩余时间引入新内容。
- **PLAN-003**：未来 7 天预测超过时间预算时减少或暂停新卡。
- **PLAN-004**：没有足够历史时使用保守默认值并明确为初始计划。
- **PLAN-005**：计划生成必须可解释，至少给出到期、目标优先级或薄弱原因。
- **PLAN-006**：用户可临时调整当天时长，但系统不删除未完成任务。

### 9.6 主动回忆和复习

- **REV-001**：答案默认隐藏，先要求用户回忆。
- **REV-002**：支持 Again/Hard/Good/Easy 四档评分。
- **REV-003**：部分关键卡支持书写或结构化作答，普通卡保留低摩擦心中回忆。
- **REV-004**：提交记录评分、耗时、是否查看提示、会话 ID 和幂等键。
- **REV-005**：使用标准 FSRS 输出下一次到期时间和算法版本。
- **REV-006**：重复提交或网络重试不得创建重复作答记录。
- **REV-007**：用户可标记卡片有误、来源有误、过难、过大或概念混淆。

### 9.7 学习干预

- **COACH-001**：连续遗忘的卡片进入 repair 建议，而不是无限重复原卡。
- **COACH-002**：相同标签下的易混卡可生成对比练习。
- **COACH-003**：复习积压时停止新增并建议缩短范围或增加时间。
- **COACH-004**：每周提供跨章节混合测试，检测脱离原文后的回忆。
- **COACH-005**：Qwen 解释必须引用发布内容；依据不足时返回无法确认。
- **COACH-006**：学习建议由规则和真实记录触发，不能只输出通用鼓励文案。

### 9.8 统计与反馈

- **INSIGHT-001**：展示学习天数、实际分钟、复习量和新卡量。
- **INSIGHT-002**：展示当前到期、未来 7 天负荷和积压情况。
- **INSIGHT-003**：按学科、章节和主题展示掌握趋势、遗忘和反复错误。
- **INSIGHT-004**：统计区分“内容已覆盖”“卡片已发布”“已加入学习”“当前掌握”。
- **INSIGHT-005**：统计不以连续打卡替代学习效果。

### 9.9 文档处理控制面

- **DOC-001**：登记原始 PDF 指纹、页数、大小、版权备注和版本。
- **DOC-002**：对大文件按章节或 20–30 页拆分并保留源页映射。
- **DOC-003**：所有 704 页必须有 terminal 状态：ready、needs_review 或 failed。
- **DOC-004**：解析、清洗、结构化、生成、审核和发布分别记录版本。
- **DOC-005**：失败任务可重试，重复执行不重复产出同版本实体。
- **DOC-006**：文档处理只通过 CLI、任务服务或独立管理工具完成，不放进学习小程序。

### 9.10 候选卡与发布

- **PUB-001**：生成结果默认 `needs_review`，不得默认发布。
- **PUB-002**：支持单张审核和低风险章节批量确认。
- **PUB-003**：剂量、毒性/禁忌、针灸操作、多教材版本等高风险卡逐张审核。
- **PUB-004**：发布包包含 schema 版本、内容版本、稳定 ID、来源和审核记录。
- **PUB-005**：后端导入发布包必须幂等，并输出创建、更新、跳过和冲突统计。

## 10. 小程序信息架构

### 一级入口

1. **今日**：当日计划、预计时长、开始学习和会话结果。
2. **学科**：7 本书、章节、内容覆盖、加入学习和来源浏览。
3. **进度**：未来负荷、学科掌握、薄弱主题和周测。
4. **我的**：Owner、目标、时间预算、登录、安全和数据管理。

复习、章节学习、来源详情和周测是从一级入口进入的任务页，不占 TabBar。

### 不出现在小程序中的内容

- MinerU 批次、上传 URL 和 API Token。
- 原始 OCR 文件树和本地绝对路径。
- Qwen Key、数据库连接和 AppSecret。
- 大文件拆分、重跑和压缩包下载操作。

## 11. 学习策略基线

初始默认值用于冷启动，不代表最终个性化参数：

- 每日预算：20 分钟，可修改。
- 新卡上限：每日 5–10 张，根据未来负荷动态降低。
- 目标记忆率：0.90，待有足够真实复习数据后评估。
- 学习顺序建议：基础理论 -> 诊断学 -> 中药学/方剂学 -> 内科学 -> 针灸学；人文可独立穿插。
- 每周至少一次混合测试。

学习策略吸收 Anki、RemNote、Quizlet、Duolingo、墨墨背单词和不背单词的公开实践，但不复制其社交、排行榜、订阅或内容市场。

## 12. 非功能需求

### 性能

- 今日页在正常网络下 P95 首次数据响应目标小于 2 秒。
- 单次复习提交 P95 服务端处理目标小于 500 ms，不含网络。
- 小程序不一次性加载超过当前页面需要的卡片和长原文。
- 内容目录使用分页和按章加载。

### 可靠性

- 所有写接口支持幂等或重复保护。
- 数据库迁移可在全新库和已有原型库执行。
- 文档批次支持断点续跑和失败重试。
- 每日备份，至少完成一次恢复演练。

### 安全与隐私

- 生产 API 仅 HTTPS。
- AppSecret、模型 Key、数据库凭据仅在服务端。
- 日志不保存完整原文、完整答案上下文或密钥。
- 只保存完成学习功能所需的最小个人资料。

### 可维护性

- API、发布包、清洗规则和算法都有版本号。
- 领域逻辑有自动化测试。
- 文档处理和小程序可以独立部署和演进。

## 13. 成功指标

### 内容指标

- PDF 页处理覆盖：704/704。
- terminal 状态覆盖：100%。
- 发布卡来源可追溯率：100%。
- 高风险发布卡人工审核率：100%。
- 重复执行产生重复实体数：0。

### 产品指标

- 用户能在 30 秒内理解当天计划并开始学习。
- 完成一轮“回忆 -> 显示答案 -> 评分 -> 下一张”的成功率为 100%。
- 复习提交重试产生重复记录数：0。
- 未来负荷超预算时新卡自动收缩。
- 薄弱点建议能指向具体卡片、主题和来源，而非泛化文案。

### 学习指标

- 记录真实完成分钟和复习表现，不只记录打开次数。
- 经过至少两周使用后评估目标记忆率、积压率和每天负荷。
- 连续遗忘卡能够进入修复流程。

## 14. 发布验收

### 本地 Alpha

- 唯一 Owner 可登录并保存档案。
- 15 张样例卡迁移后历史和来源不丢失。
- 内容目录、enrollment 和个人 review state 已分离。
- 小程序完成每日计划和一次幂等评分。

### Content Alpha

- 7 本书全部进入文档清单。
- 每本至少一个章节完成解析、生成、审核和发布端到端。
- 704 页全量任务启动后可恢复、可重试、可输出覆盖报告。

### Personal Beta

- 704 页全部达到 terminal 状态。
- 用户可按章节逐步加入学习，不出现全量卡第一天到期。
- 连续使用两周后可查看负荷、薄弱点和周测结果。

### Production-ready Personal

- PostgreSQL、Alembic、HTTPS、微信登录、备份恢复和真机验证完成。
- 默认开发 Token 不再作为生产认证。
- 关键接口和领域逻辑有自动化测试与可观察日志。

## 15. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| OCR 把药名、剂量、穴位识错 | 高风险错误进入学习 | 高风险逐张审核、可疑词规则、来源对照 |
| 生成卡过多 | 复习负荷失控 | 发布与加入学习分离、未来负荷限制 |
| 多教材版本混淆 | 错误归纳 | 版本字段、版本化卡片类型、禁止合并 |
| 自评分偏差 | FSRS 信号失真 | 部分书写题、耗时和提示信号、周测校验 |
| 当前原型缺少迁移 | 数据丢失 | 先建立 Alembic 和迁移测试 |
| 外部 API 限额或失败 | 全量处理停滞 | 小批次、manifest、重试、成本记录 |
| 旧文档驱动错误范围 | 重复或冲突开发 | 以 `docs/superpowers/` 为唯一新基线 |

## 16. 需求变更规则

新增需求必须说明它属于文档处理控制面、后端领域或学习小程序中的哪一层。无法归类的功能先不实现；跨层需求必须同步修改本 PRD、对应设计文档和实施计划。
