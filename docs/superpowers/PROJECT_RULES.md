# wxzy 项目规则

状态：Active
适用对象：后续所有模型、开发者和自动化任务

这些规则采用 Superpowers 风格：先理解边界，再写规格和计划；每个任务都必须有可检查的完成条件；没有证据时不能声称完成。

## 1. 工作协议

### SP-001 先读再改

修改前必须读取：

- `AGENT.md`。
- `docs/superpowers/README.md`。
- 当前任务对应的 PRD、设计文档和实施任务。
- 将要修改文件的现状和关联测试。

### SP-002 规格先于跨边界代码

涉及两个或以上子系统、数据模型、接口、卡片状态、认证或学习算法的改动，必须先更新规格或设计文档，再写代码。单文件的明显缺陷修复可以直接处理，但仍需补验证。

### SP-003 任务必须可验收

每个实施任务必须写明：目标、范围外内容、依赖文件、输入输出、验证命令、验收条件和已知风险。任务粒度以一次专注工作可以完成为准，不把“完成整个系统”作为一个任务。

### SP-004 证据优先

报告“已完成”只能依据测试输出、接口响应、编译结果、质量报告或可复现的文件检查。静态检查通过不等于运行时流程通过；样本通过不等于全量 PDF 通过。

### SP-005 小步提交

每次改动保持单一主题，避免顺手重构无关模块。生成物、缓存、临时 PDF、数据库和密钥不进入提交。跨阶段任务应留下可恢复的中间状态。

## 2. 架构边界

### SP-010 文档处理和小程序必须分离

`tools/` 及未来的文档任务服务属于控制面；`miniprogram/` 属于用户运行面。小程序不得导入 MinerU、Qwen、PyMuPDF、数据库驱动或任何服务端密钥。

### SP-011 内容目录和学习状态必须分离

发布卡是共享内容目录中的事实对象；用户进度、到期时间、评分和学习计划是用户状态。发布卡不能因为被导入就自动让所有卡片在用户当天到期。

### SP-012 服务器分层

- `routers/` 只负责 HTTP 参数、鉴权依赖和错误映射。
- `schemas/` 只负责输入输出契约，不承载数据库查询。
- `services/` 或领域模块承载业务规则和事务边界。
- `models/` 只描述持久化模型和关系。
- 调度算法必须是纯函数或独立领域服务，可单独测试。

### SP-013 工具脚本可恢复

长任务必须有 manifest、稳定任务 ID、阶段状态、重试次数、错误摘要和输出位置。重复执行不能重复生成同一版本内容，也不能覆盖原始输入。

## 3. 单用户身份和数据规则

### SP-020 单用户不是无用户

系统只允许一个 Owner，但数据库仍保留 `user_id`/`owner_id` 外键。这样可以正确保存个人学习状态、支持多设备同步、备份和未来迁移，而不引入多用户界面。

### SP-021 认证分环境

本地开发可以使用固定 Bearer Token；正式小程序使用 `wx.login`、服务端 Session 和服务端保存的 AppSecret。Token、OpenID、AppSecret 和模型密钥不能进入小程序代码包或日志。

### SP-022 个人资料最小化

默认只保存 OpenID、时区、学习目标、每日时间、科目优先级和可选昵称。不得为了“丰富画像”收集与学习无关的手机号、健康信息或完整微信资料。

## 4. 内容和卡片规则

### SP-030 来源不可断链

每个内容块和发布卡必须能回溯到：文档、文档版本、PDF 文件页、印刷页（若存在）、章节、内容块 ID 和原文摘录。页码不确定时必须标记不确定，不得猜测。

### SP-031 候选不能直接复习

状态流转必须是 `generated -> needs_review -> approved/rejected -> published`。`approved` 表示内容审核通过；`enrolled` 才表示进入某个用户的学习范围。

### SP-032 高风险内容单独复核

方剂剂量、药名、毒性和禁忌、针灸定位和操作、证型与代表方、多教材版本、古文方歌等内容，不能仅凭模型置信度自动发布。

### SP-033 生成器不得截断全书

任何 `max_chars`、`max_pages` 或批次限制都必须通过显式分块遍历处理，并记录遗漏和重叠范围。单次只处理输入前段不算全量完成。

## 5. 后端和数据库规范

### SP-040 数据库必须可迁移

生产环境不得依赖 `Base.metadata.create_all()` 建表。使用 Alembic（或等价迁移工具）管理版本；SQLite 只用于本地快速验证，PostgreSQL 是目标运行库。

### SP-041 时间和标识

数据库统一保存 UTC 时间，接口明确时区转换。外部内容 ID、文档版本 ID、任务 ID 和幂等键必须稳定、可重复计算或明确生成来源。

### SP-042 接口契约稳定

新业务接口使用 `/api/v1`；健康检查可保留 `/health`。错误响应至少包含稳定 `code`、用户可理解的 `message` 和 `request_id`。改变字段含义必须增加版本或迁移说明。

### SP-043 写操作幂等

导入、批准、加入学习和提交复习都必须支持重复请求保护。复习按钮重复点击、网络重试和客户端超时不能产生重复 ReviewAttempt。

### SP-044 标准算法优先

FSRS 使用经过验证的标准实现和显式 `algorithm_version`。当前 `fsrs-v1` 轻量实现只能作为迁移过渡，不得把它描述为已经完成标准 FSRS 评测。

## 6. 语言与代码风格

### SP-045 Python

- 目标运行时为 Python 3.12。
- 使用 4 空格、UTF-8、Ruff format，目标行宽 100；机械格式化与行为修改分开提交。
- 公共函数、领域服务、Pydantic schema 和复杂集合必须有类型标注。
- 文件路径使用 `pathlib.Path`，时间使用 timezone-aware UTC，不混用 naive datetime。
- 不使用 bare `except`；异常必须被转换、记录或继续抛出，不能静默吞掉。
- API 服务使用结构化 logger，不使用 `print`；CLI 可以输出进度，但不得输出密钥和完整签名 URL。
- `Any` 只用于不可信外部边界，进入领域层前必须校验成明确 schema。
- import 按标准库、第三方、本项目分组；不通过运行时修改 `sys.path` 组织模块。

### SP-046 原生小程序 JavaScript

- 沿用仓库现有的 2 空格、无分号、CommonJS 风格，直到有单独规格批准构建工具迁移。
- 默认使用 `const`，确需重新赋值时使用 `let`，不使用 `var`。
- 网络和异步流程使用 Promise/`async`/`await`，每个写操作都必须有提交锁和失败恢复。
- 页面事件以 `on` + 明确动词命名；API 函数以 `get/list/create/update/submit` 等动作命名。
- 不把复杂表达式、排序、过滤或算术堆进 WXML；在 JS 中计算后通过最小 `setData` 更新。
- 不使用可选但未经基础库验证的语言/运行时特性；新增能力先核对微信兼容性。

### SP-047 WXML/WXSS/JSON

- WXML 只负责结构和简单绑定；循环使用稳定 `wx:key`，事件 dataset 只传轻量标识。
- WXSS 以页面/组件前缀隔离类名；关键尺寸使用稳定约束，长文本允许换行。
- 不依赖 `:root`、CSS 自定义属性、浏览器专属选择器或未经验证的布局特性。
- JSON 必须是严格 JSON、2 空格格式、无注释；页面标题表达页面用途，不统一写成无差别标题。

### SP-048 SQL、Schema 与命名

- 表名使用复数 snake_case，主键统一 `id`，外键为 `<entity>_id`。
- 业务唯一性由数据库约束保证，不能只靠应用先查后写。
- API 字段使用 snake_case；枚举值使用稳定小写 snake_case；错误 code 使用大写 snake_case。
- JSONB/JSON 字段必须有 Pydantic/JSON Schema 契约，不能变成无约束杂物箱。
- migration 名称描述业务变化；迁移内不导入会随应用演进的领域服务。

### SP-049 依赖与注释

- 优先使用标准库和仓库已有依赖；新增依赖必须说明维护状态、许可证、体积和替代方案。
- FSRS、PDF 解析、迁移和安全相关能力优先使用成熟库，不重复手写成熟算法。
- 注释解释原因、约束和非显然风险，不复述代码动作。
- TODO 必须关联实施任务 ID；无 owner/任务的 TODO 不进入主分支。

## 7. 文档处理工程规范

### SP-050 原始输入不可变

原 PDF、原始 MinerU 压缩包和原始 `full.md` 只读保存。清洗、结构化、候选卡和发布导出都写入版本化输出目录，不覆盖上一阶段。

### SP-051 分块优先按语义

优先按书籍章节和页边界分块；为了满足外部服务大小限制再做 20–30 页物理拆分。每个拆分块必须包含原书页范围和映射表。

### SP-052 质量门禁先于卡片

解析阶段必须检查页覆盖、文本/表格结构、页码映射、OCR 可疑词、图片完整性和章节顺序。质量不通过的块只能进入待复核区，不能喂给正式卡片生成。

### SP-053 书籍模板显式化

每类书籍的卡片类型、字段要求、风险词和审核规则写成配置或模板，不把所有书籍硬塞进一个正则提取器。

## 8. 小程序工程规范

### SP-060 小程序只通过 API 层通信

页面不得散落 `wx.request`；请求、鉴权、超时、错误映射和重试集中在 `services/`。页面只处理展示状态和用户事件。

### SP-061 状态完整

每个网络页面必须有加载、空数据、失败、重试、无权限、完成和重复点击防护状态。页面切换或重复进入时不得把旧请求结果写入新页面。

### SP-062 兼容微信运行时

使用原生 `App/Page/Component`、WXML/WXSS 支持的语法和 `rpx`。不使用 `window`、DOM、浏览器 `localStorage`、未验证的 CSS 自定义属性或 Web 专属 API。

### SP-063 数据按需加载

不把整本 PDF、长篇 OCR、完整问答上下文或大量卡片一次性写入 `data`。书籍、章节、来源和学习队列按需分页。

### SP-064 视觉和文案边界

页面是学习工具，不是医疗诊断、营销页或社区。按钮、评分、筛选和加载状态必须可见且可理解；医学内容必须保留学习边界和来源。

## 9. 测试和质量门禁

### SP-070 后端

必须逐步建立：`pytest` 单元/集成测试、Ruff lint/format、类型检查、迁移升级测试、API 契约测试和真实 SQLite/PostgreSQL 冒烟。

### SP-071 文档流水线

必须有：小 PDF fixture、页码映射测试、OCR 清洗测试、HTML 表格解析测试、稳定 ID 测试、重复运行测试、候选 schema 校验和质量报告快照。

### SP-072 小程序

至少执行 Node 语法检查、严格 JSON 解析、开发者工具编译、模拟器网络冒烟和人工四页流程。跨页面状态变化必须有真实接口验证。

### SP-073 风险比例

只读 UI 改动做 focused smoke；数据模型、认证、复习算法、导入和跨模块契约改动必须扩大测试范围。不能用“没有测试基础设施”作为跳过验证的理由，先补最小设施。

## 10. 安全、隐私、版权和运维

- 所有生产 API 使用 HTTPS 和合法域名。
- 原始 PDF 和 OCR 产物默认私有，不公开整本受版权保护内容。
- 日志只记录任务 ID、耗时、状态和脱敏错误，不记录密钥或完整原文。
- 数据库、内容目录和学习记录可备份、恢复和删除。
- 医疗相关问题只提供文献学习解释，不输出个体诊断、处方或替代医生建议。

## 11. Definition of Done

一个任务只有同时满足以下条件才可标记完成：

1. 需求、设计或假设已经写清楚。
2. 代码和文档边界没有越权。
3. 正常、空、失败、重试和重复操作已验证。
4. 相关测试、静态检查或开发者工具编译通过。
5. 数据迁移、回滚、日志和安全影响已说明。
6. 交付报告明确写出已完成、未完成和剩余风险。
