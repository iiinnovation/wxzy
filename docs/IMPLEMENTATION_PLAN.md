# 中医学习小程序实施方案

> 状态：Legacy。本文保留作早期方案；当前可执行任务以 `docs/superpowers/plans/2026-07-22-wxzy-implementation-plan.md` 为准。

## 1. 方案说明

这份文档把 [产品规划](PRODUCT_PLAN.md) 转换成可执行的实施顺序，回答三个问题：

1. 用户需要准备什么资源和决策。
2. 我可以负责哪些设计、编码、测试和部署工作。
3. 每个阶段完成到什么程度，才进入下一阶段。

当前默认前提：产品先服务单个用户，使用微信小程序原生框架；PDF、向量模型和 Rerank 模型部署在阿里云；Qwen 使用线上 API；第一版以学习、检索和复习为主，不做诊疗业务。

## 2. 先做的事情

开始编码前，先完成以下五项。它们是项目的第一道门槛：

1. 确认 5～10 本有权使用的代表性 PDF，并放入一个待处理目录。
2. 注册或确认微信小程序账号、AppID、开发者身份和微信开发者工具可用。
3. 确认阿里云服务器、域名、HTTPS 和对象存储方案。
4. 开通 Qwen 线上 API，并在服务端安全保存 API Key、模型名称和额度信息。
5. 从真实学习需求中整理 30～50 个问题，并标注正确文档、章节和页码，作为检索评测集。

在这五项完成前，不建议大量开发页面。PDF 结构和真实问题会直接影响分块、检索、引用和学习卡片设计。

## 3. 责任分工

### 3.1 用户负责准备和决策

- 提供有权使用的 PDF 样本，说明哪些文献优先级最高。
- 决定首批学习范围，例如理论、伤寒、金匮、方剂、药物或医案。
- 注册并管理微信小程序、阿里云、OSS、域名和 Qwen 账号。
- 提供不包含密钥的环境信息，例如地域、操作系统、域名和可用磁盘。
- 对 OCR 质量、检索结果、模型解释和复习题目进行最终学习判断。
- 确认产品文案、隐私边界和是否允许保存学习记录。
- 使用真实问题进行验收，指出“看似相关但不是正确依据”的结果。

不要在聊天、代码仓库、截图或日志中发送 AppSecret、Qwen API Key、SSH 私钥、数据库密码或个人隐私数据。凭据应通过服务器环境变量或密钥管理服务配置。

### 3.2 我负责的工作

- 把需求拆成接口、数据模型、页面状态和可验收任务。
- 建立微信小程序项目骨架和符合 `AGENT.md` 的代码结构。
- 设计并实现后端 API、任务队列、文献导入、复习算法和权限边界。
- 实现 PDF 文本抽取、OCR 兜底、章节识别、分块、向量化和索引流程。
- 集成向量模型、Rerank 模型和 Qwen，建立可追溯引用的问答链路。
- 实现文献阅读、检索、卡片、复习和问答页面。
- 编写单元测试、接口测试、检索评测脚本和部署文档。
- 在用户提供环境信息后协助部署、排错、备份和性能验证。

### 3.3 共同完成的工作

- 评估 PDF 解析和 OCR 是否足以支撑学习。
- 为检索结果制定“正确、部分正确、错误、无依据”判定标准。
- 审核模型生成的卡片、题目和解释。
- 决定哪些内容进入正式复习库，哪些只作为临时问答。
- 根据实际学习体验调整卡片结构、复习间隔和每日任务量。

## 4. 用户需要准备的资源

### 4.1 微信小程序

- 微信公众平台小程序账号和 AppID。
- 账号主体、管理员和开发者权限。
- 本地微信开发者工具。
- 用于真机预览的微信账号。
- 小程序合法业务类目和隐私保护指引准备。

自用阶段可以先使用开发版或体验版，不必立即做公开运营。但真机和正式版本通常需要配置合法的 HTTPS 业务域名，并遵守微信平台的隐私和内容要求。

### 4.2 阿里云

- 一台可通过 SSH 管理的 Linux 服务器。
- 一个稳定域名；若部署在中国大陆地域，提前确认备案要求和周期。
- HTTPS 证书，建议通过 Nginx 统一终止 TLS。
- OSS Bucket，用于保存 PDF、页面图片和 OCR 中间文件。
- 数据库备份位置，不能只依赖服务器本地磁盘。
- Qwen 在线 API 账号、模型额度和调用区域。

### 4.3 样本文献

第一批不要直接上传全部 PDF，先准备能覆盖不同情况的样本：

- 数字文本 PDF。
- 扫描版 PDF。
- 繁体字或竖排古籍。
- 章节结构清晰的现代教材。
- 含方剂、药物表格或医案的文献。

每份样本附带书名、作者、版本、优先级、版权备注和预期页数。原始 PDF 只保留一份不可变版本，后续处理结果用版本号区分。

### 4.4 评测问题集

每个问题至少记录：

```text
问题
期望命中的文档
期望章节
期望页码或页码范围
问题类型
允许的多个答案（如有）
是否要求原文引用
```

问题类型建议覆盖：概念解释、方剂组成、方剂鉴别、证候辨析、药物功效、医案推理和古文转述。

## 5. 推荐的第一版技术基线

### 5.1 应用层

- 小程序：微信原生 `WXML + WXSS + JS + JSON`。
- 后端 API：Python FastAPI。
- 异步任务：Redis + Celery 或 RQ。
- 数据库：PostgreSQL + pgvector。
- 文件：阿里云 OSS。
- 反向代理：Nginx。
- 部署：Docker Compose 起步，后续再按负载拆分。

这个组合适合单用户和初始验证，组件数量较少，迁移成本低。第一版不提前引入 Kubernetes、复杂微服务或独立向量数据库。

### 5.2 模型层

向量和 Rerank 模型先做可替换接口，不在业务代码中绑定具体模型名。建议准备一个成熟的中文基线进行评测，例如 BGE 系列；如果阿里云 GPU 资源和中文效果更适合，也可以评估 Qwen Embedding/Reranker 系列。

模型选择依据不是参数量，而是以下评测：

- 中医术语和古籍词语的召回。
- 方剂名、药名和证候名的精确匹配。
- 繁体字、异体字和 OCR 噪声下的稳定性。
- 单次检索延迟和显存占用。
- 在 30～50 个问题集上的 Top-k 命中率。

Qwen 只负责有依据的解释、题目生成和答案反馈。API 调用必须封装在服务端，并记录模型版本、耗时、错误和费用统计。

### 5.3 服务器基线

最终规格以模型压测为准。开始评估时可按下列基线准备：

- API、数据库、Redis、任务服务：4～8 vCPU、16～32 GB 内存、100 GB 以上 SSD。
- 向量和 Rerank 推理：优先使用 16 GB 以上显存的 GPU；没有 GPU 时可先用 CPU 小规模验证，但不作为批量生产方案。
- PDF 和 OCR 临时空间：按原始 PDF 总量至少预留 2～3 倍空间。
- 备份：数据库和处理结果至少保留一份异地或 OSS 备份。

不要把模型、数据库和大规模 OCR 临时文件都堆在系统盘上。磁盘、显存和 API 额度都需要监控。

## 6. 系统实施顺序

### 阶段 0：需求和样本冻结

**目标**：让开发依据真实文献和真实学习问题进行。

我完成：

- 固化 MVP 页面、接口和数据实体。
- 建立评测问题格式和人工判定规则。
- 设计文献处理状态机和失败重试状态。
- 设计“候选卡片”和“正式卡片”的审核边界。

用户完成：

- 提供 5～10 本样本 PDF。
- 标注书名、作者、版本和使用权限。
- 提供 30～50 个真实问题和期望出处。
- 确认第一批学习主题及优先顺序。

阶段完成标准：样本和问题集可以在不依赖模型猜测的情况下进行人工评测。

### 阶段 1：项目骨架和本地开发环境

**目标**：所有服务可以本地启动，并能通过最小 API 联通。

我完成：

- 建立小程序、后端、任务服务和基础配置目录。
- 建立 PostgreSQL、pgvector、Redis 的本地开发配置。
- 建立环境变量模板、数据库迁移和健康检查接口。
- 建立统一错误格式、日志格式和请求追踪 ID。
- 建立基础 CI 检查：格式、静态检查、单元测试和构建。

用户完成：

- 安装微信开发者工具和项目所需本地运行环境。
- 提供不含密钥的阿里云系统和域名信息。

阶段完成标准：本地能启动 API、数据库和小程序，健康检查通过，密钥不进入仓库。

### 阶段 2：文献导入和解析

**目标**：把 PDF 变成带页码、章节和质量状态的可检索文本。

我完成：

- PDF 上传、校验、对象存储和任务状态。
- 数字 PDF 文本抽取。
- 扫描 PDF OCR 兜底。
- 页眉页脚清理、乱码标记和页面级原文保存。
- 章节标题识别、文本分块和元数据保存。
- 失败重试、断点续跑和人工查看处理结果的页面/API。

用户完成：

- 抽查每种样本的若干页，标记不可接受的 OCR 错误。
- 确认哪些章节可以作为第一批学习内容。

阶段完成标准：每个片段都能回到原 PDF 的文档名、章节和页码；低质量片段被标记，而不是静默进入问答。

### 阶段 3：检索、Rerank 和 Qwen 问答

**目标**：问答回答有依据、可引用、可评测。

我完成：

- 向量模型服务适配和批量向量化任务。
- 基于 pgvector 的语义检索。
- 精确词、书名、方剂名和药名的关键词检索。
- 候选合并、去重和 Rerank。
- Qwen 上下文拼接、提示词、超时、重试和限流。
- 来源片段、章节、页码和模型版本返回。
- “依据不足”和“模型无法确定”的安全状态。

用户完成：

- 对 30～50 个评测问题逐题判断检索和引用是否正确。
- 指出相似方剂、药物和证候的误召回。
- 确认回答语气和古文解释深度。

阶段完成标准：评测问题的首要依据可追溯；错误答案不会被包装成确定事实；模型密钥只存在服务端。

### 阶段 4：小程序阅读和学习功能

**目标**：用户可以完成从文献阅读到知识卡片的基本操作。

我完成：

- 文献列表、章节树、分页阅读和文献内搜索。
- 问答页面、引用展开和回到原文。
- 手动创建、编辑、归档知识卡片。
- 题目、答案、解析和来源片段展示。
- 加载、空状态、失败、重试和长文本布局。

用户完成：

- 用手机完成一次完整阅读和卡片创建。
- 确认文字大小、古文换行、页码跳转和复制体验。

阶段完成标准：不依赖后台管理界面，用户可以从小程序完成一次“查找、阅读、创建卡片”。

### 阶段 5：艾宾浩斯复习闭环

**目标**：学习卡片能够进入每日复习并根据结果重新排期。

我完成：

- 今日到期队列。
- 先答题、再显示答案和解析。
- 掌握程度记录和幂等提交。
- 当天、1 天、3 天、7 天、15 天、30 天的初始间隔算法。
- 复习历史、连续正确和遗忘次数。
- 卡片状态与文献来源关联。

用户完成：

- 连续使用至少一周，记录复习量是否合理。
- 标记“题目过难、答案不完整、来源不对或间隔不合适”的卡片。

阶段完成标准：刷新、重新进入和重复点击都不会造成重复记录；不同掌握结果会得到不同下一次复习时间。

### 阶段 6：AI 辅助学习和卡片审核

**目标**：让 Qwen 辅助学习，而不是无审核地生产知识。

我完成：

- 从章节或片段生成候选卡片。
- 古文转述、分层解释、对比题和答案批改。
- 候选卡片审核、编辑、批准、拒绝和来源查看。
- 模型输出风险标记和引用检查。
- Prompt 版本化和生成结果可追踪。

用户完成：

- 审核首批候选卡片。
- 选定最有价值的题型和解释方式。
- 明确哪些主题必须人工编辑后才能进入复习。

阶段完成标准：未审核内容不会出现在正式复习队列；每张正式卡片都有来源和可查看的生成记录。

### 阶段 7：阿里云部署和稳定性

**目标**：在真实手机和真实服务器上稳定运行。

我完成：

- Docker Compose 或等价部署配置。
- Nginx、HTTPS、域名、健康检查和进程重启。
- 数据库迁移、备份和恢复演练。
- 日志脱敏、服务监控、任务失败告警和费用统计。
- 微信业务域名配置说明。
- 生产环境与开发环境隔离。

用户完成：

- 配置服务器、域名、证书、OSS 和 Qwen 环境变量。
- 在微信公众平台配置合法域名和隐私保护资料。
- 用真实手机验收主要流程。

阶段完成标准：服务器重启后服务可恢复；数据库可从备份恢复；网络超时、任务失败和模型限流都有可理解的用户提示。

## 7. API 和数据实现优先级

第一批优先实现以下接口，后续再扩展：

```text
GET  /health
GET  /documents
GET  /documents/{id}/chapters
GET  /chapters/{id}/content
POST /search
POST /chat

GET  /review/today
POST /review/{card_id}/answer
GET  /cards
POST /cards
PATCH /cards/{id}

POST /admin/documents/upload
GET  /admin/import-tasks/{id}
POST /admin/cards/{id}/approve
```

最小数据表为：`documents`、`document_chunks`、`import_tasks`、`knowledge_cards`、`review_records`、`chat_records` 和 `users`。表结构需保留来源、版本和算法版本，避免以后无法解释历史结果。

## 8. 质量门槛

### 8.1 文献质量

- 数字 PDF：文本能够按页保存，页码与原 PDF 一致。
- OCR PDF：抽查结果能区分可用、需修正和不可用。
- 方剂和药物列表不允许因清洗被静默合并或删除。
- 文档版本变化后重新处理，不能覆盖旧处理结果。

### 8.2 检索质量

对评测问题记录：

- Top-1、Top-3 和 Top-5 是否出现正确片段。
- Rerank 是否把正确片段排到前面。
- 引用页码是否正确。
- 是否出现相似概念误召回。
- 检索不到时是否诚实返回依据不足。

MVP 不预设脱离样本的绝对准确率数字，先建立基线，再根据真实问题集设定目标。任何指标都不能替代用户对关键文献的人工核对。

### 8.3 学习质量

- 用户能在 1 分钟内理解当前要复习什么。
- 题目显示答案前确实要求主动回忆。
- 复习结果会改变下一次到期时间。
- 题目、解析和来源之间没有断链。
- 每日任务量可控，不因自动生成卡片而无限膨胀。

## 9. 安全、隐私和运维准备

### 必须完成

- 所有 API 使用 HTTPS。
- 小程序 AppID、用户标识和服务端登录态分离管理。
- Qwen、数据库、OSS 和服务器凭据使用环境变量或密钥管理服务。
- OSS 原始 PDF 默认私有，使用短期签名地址或服务端代理访问。
- 日志不保存完整 PDF、完整问答上下文和密钥。
- 数据库每日备份，并至少做一次恢复演练。
- 文献访问按用户和文档权限校验，不能只依赖前端页面隐藏。
- 公开部署前检查微信隐私保护指引和文献版权范围。

### 暂不做

- 多租户复杂权限。
- 公共文献分享。
- 用户之间的内容流转。
- 面向患者的症状收集和健康档案。

## 10. 推荐的第一次执行清单

用户先完成：

1. 选出 5～10 本样本文献并打包整理。
2. 选出 30～50 个真实问题，尽量包含方剂、药物、证候和医案。
3. 确认小程序 AppID、开发者工具和微信真机账号。
4. 提供阿里云服务器的系统、地域、GPU、磁盘和域名状态。
5. 确认是否使用 OSS、数据库服务和 Qwen 在线 API。

我随后完成：

1. 把样本和评测问题转换成可执行的数据格式。
2. 输出最终技术选型和服务器部署清单。
3. 建立本地项目骨架、数据库迁移和健康检查。
4. 先实现文献导入和检索评测，不急着做完整 UI。
5. 用评测结果决定向量模型、Rerank 模型和分块参数。

## 11. 关键决策点

以下决策会影响工期和成本，需要在进入对应阶段前确认：

1. 第一批学习主题优先理论、方剂、药物还是医案。
2. PDF 是否全部属于个人可使用范围，是否需要做文档级访问限制。
3. 阿里云服务器是否有可用 GPU，以及显存大小。
4. 是否接受第一版仅支持单个固定用户。
5. 是否需要在第一版支持扫描古籍和繁体竖排 PDF。
6. Qwen 更重视低成本、低延迟还是复杂解释质量。
7. 每日默认复习量和卡片审核方式。

只要这些问题没有改变产品边界，就可以先按本方案推进；它们不应阻塞文献解析和检索基线的建立。
